
## Plan: Offline Demo Mirror

Build a fully isolated Try Demo experience under `lib/demo/` that mirrors the real order/account/funds/history flows while keeping production routes and feature behavior untouched. The demo route tree will be `/demo/...`, all demo state and API behavior will live in Hive-backed services under `lib/demo`, and only live market feed updates will continue to come from the existing websocket feed. Existing UI widgets should be reused wherever they can be safely scoped to demo blocs; demo-specific wrappers will be added only where current widgets hard-code real route prefixes or direct REST calls.

**Steps**
1. Capture the isolation contract in `lib/demo/readme.md` before code changes: demo code stays under `lib/demo`; outside changes are limited to Try Demo buttons and route registration/redirect mapping; existing real routes `/home`, `/sport`, `/account`, `/result`, `/inplay`, `/multimarkets` must continue to work exactly as they do today.
2. Add demo route definitions: create `lib/demo/routing/demo_routes.dart`, `lib/demo/routing/demo_route_navigation_helper.dart`, and `lib/demo/routing/demo_route_mapper.dart`; minimally update `lib/routing/app_routes_constants.dart`, `lib/routing/app_router.dart`, and `lib/routing/route_navigation_helper.dart` to register `/demo/home`, `/demo/inplay`, `/demo/inplay/today`, `/demo/inplay/tomorrow`, `/demo/sport/:sportId/:screenType`, `/demo/sport/:sportId/:screenType/event/:eventId/:inplay/:premium`, `/demo/multimarkets`, `/demo/result`, `/demo/result/:tab`, `/demo/account`, `/demo/account/:section`, and `/demo/account/my-bets/:tab`.
3. Add route mapping for reused widgets: because many existing widgets call `GoToRoute.*` and produce real paths, `DemoRouteMapper` should convert real-path navigation to demo-prefixed paths only while a demo session is active. This is route-related and keeps existing widgets reusable without changing their source.
4. Add demo shell wrappers under `lib/demo/presentation/`: `DemoMobileMainView` and `DemoDesktopMainTab` should mirror the current mobile/desktop shells but normalize the `/demo` prefix for selected tab/index logic and use demo route helpers for bottom nav and desktop tab links. Reuse `CustomSportAppBar`, `MarqueeNews`, account menus, bet slip/open-bets widgets, and sports/account screens where scoped demo blocs make them safe.
5. Add Try Demo entry points: minimally update `lib/screens/mobileScreens/home_screen_mobile_appbar.dart` and `lib/screens/deskopView/mainTabView/desktop_appbar_login.dart` to show `Try Demo` beside Login/Sign up. The button calls `DemoBootstrap.start(context)` from `lib/demo/bootstrap/demo_bootstrap.dart`, seeds/reset demo Hive state, activates demo session, then navigates to `/demo/home`.
6. Implement demo Hive storage under `lib/demo/data/`: use a lazily opened box such as `dmx_demo_state_v1`; store JSON-compatible maps/lists, not new Hive adapters, so no model typeId conflicts or generated code are needed. Keep keys for session, credentials, profile, account balance, casino balances, activity logs, fav stakes, sports categories, competitions/events, odds/bookmaker/fancy snapshots, open orders, current bets, bet history, P&L statements, player P&L, runner P&L, fancy book, and casino history.
7. Seed complete demo state: create `demo_seed_data.dart` with a local user (`role: client`, demo username/password, userId/accountId), profile fields compatible with `UserAccountDetails.fromJson` including `users: []`, `history: []`, and `casinoAccount: []`; seed main balance/exposure/pnl, casino wallets, activity logs, fav stakes/default stake, sample open orders, settled order history, account statement rows, P&L rows, sports categories, competitions, events, and market snapshots.
8. Build `DemoOrderEngine`: on order placement, validate stake/price/min/max/balance, generate local order IDs, calculate back/lay liability, update open orders/current bets/history, recalculate exposure/balance/pnl/runner PL/fancy book, append account statement rows, emit demo account/order messages, and persist all affected state atomically to Hive.
9. Build demo API/bloc layer entirely under `lib/demo/blocs/` and `lib/demo/repositories/`. For existing widgets, provide demo implementations as `BlocProvider<ExistingBlocType>` by implementing the public bloc interface while extending `Bloc<ExistingEvent, ExistingState>`, so unchanged widgets still resolve `context.read<FetchOpenOrdersBloc>()`, `context.read<SendOrderBloc>()`, etc. without using the production constructors.
10. Cover repository-backed modules with demo equivalents: `SendOrderBloc`, `FetchOpenOrdersBloc`, `FetchCurrentBetsBloc`, `FetchOrdersBloc`, `FetchPlByOrdersBloc`, `FetchAllBetsOrdersBloc`, `FetchPlayerBetHistoryBloc`, `FetchPlayerProfitAndLossBloc`, `FetchOddsRunnerPLBloc`, `FetchBMRunnerPLBloc`, `FetchFancyRunnerPLBloc`, `FetchFancyBookBloc`, `FetchUserAccountDetailsBloc`, `FetchUserActivityLogsBloc`, `FetchCGBalanceBloc`, `AddCGMoneyBloc`, `RecallCGBalanceBloc`, `FetchCGHistoryBloc`, `FetchFavStakeBloc`, `AddFavStakeBloc`, `AddFavouriteEventsBloc`, `RemoveFavouriteEventsBloc`, and `ChangePasswordBloc`.
11. Cover current direct-REST modules with demo equivalents because they do not use repositories today: `FetchSportsCategoryBloc`, `FetchCompetitionWithEventsBloc`, `FetchSportEventsBloc`, `FetchOnlyInplayCountsBloc`, `FetchODDSDataBloc`, `FetchBookMakerBloc`, `FetchFancyDataBloc`, and `FetchScoreBoardBloc`. These must read Hive fixtures/snapshots at runtime; no REST call should run from demo routes.
12. Keep only market feed online: reuse the existing websocket event feed classes (`SignalREventListenerBloc`, `SubscribeMultiEventsSignalRBloc`, `SignalRODDSDataBloc`, `SignalRBMDataBloc`, `SignalRFancyDataBloc`, and `MultiEventsSignalRDataBloc`) inside the demo scope. Incoming odds/bookmaker/fancy websocket ticks patch `DemoMarketStore` snapshots and update UI; if a fixture event has no live tick, keep showing the seeded Hive market snapshot so the demo remains usable offline.
13. Add demo account streams under `lib/demo/streams/`: emit account balance, exposure, order messages, and runner P&L from `DemoOrderEngine` into demo versions of `SignalRAccountDataBloc`, `SignalRMessageBloc`, `OddsProfitLossBloc`, `BmProfitLossBloc`, and `FancyProfitLossBloc`. Do not connect `SignalRHubListenerBloc` to the authenticated production listener for demo accounts.
14. Add `DemoBlocScope`: a single wrapper under `lib/demo/presentation/demo_bloc_scope.dart` that provides all demo blocs/repositories above the demo ShellRoute. This wrapper is the dependency boundary that makes existing widgets read demo state while real routes keep reading production providers.
15. Add demo logout/reset behavior: demo logout clears only `dmx_demo_state_v1` session keys and demo streams, leaves production Hive/token state alone, routes back to `/home`, and calls the same reset-style events in the scoped demo blocs.
16. Keep production auth untouched: do not rely on `SaveTokenBox` for demo auth because `UserAuthChangesBloc` verifies tokens against the real auth API and may clear invalid demo tokens. Demo auth state should come from `DemoUserAuthChangesBloc` inside `DemoBlocScope`; production `UserAuthChangesBloc` remains unchanged.
17. Add focused tests where practical: unit-test `DemoOrderEngine` for back/lay order creation, insufficient balance, exposure calculation, P&L rows, current/open/history lists, and casino recall/add-money behavior; widget smoke-test `DemoBlocScope` to confirm existing account/open-bets widgets render seeded data.
18. Manual QA pass: use Try Demo on mobile and desktop, place odds/bookmaker/fancy orders, confirm open bets/current bets update, account balance/exposure changes, activity/history/P&L/funds screens show Hive-backed data, route URLs stay under `/demo/...`, real login/signup/production routes still behave normally, and no demo action calls production REST/order/auth endpoints.

Search demo runtime code for blocked endpoint usage: no `AuthApiConstants.login`, `AuthApiConstants.varifyToken`, `OrdersApiConstants.order`, `OrdersApiConstants.openOrder`, `CGApiConstants.addMoney`, or direct `SportsApiConstants.*` REST calls from demo blocs; websocket feed endpoints are allowed only for market feed.

5. Exit demo and verify production `/home`, login, signup, protected account redirect, and real order/account providers are unchanged.

**Decisions**
- Demo uses its own `/demo/...` route tree instead of reusing production protected routes.
- Demo runtime data uses Hive maps/lists and existing model `fromJson` factories to avoid new Hive adapters and codegen.
- Demo auth does not write fake tokens into `SaveTokenBox`; a scoped `DemoUserAuthChangesBloc` supplies the logged-in state for demo widgets.
- Existing UI should be reused, but existing route wrappers are replaced by demo wrappers where `/demo` prefix handling is needed.
- Only the live market websocket feed remains online; all order/account/funds/history/profile/favorites/scoreboard/sports REST-style interactions are Hive-backed in demo.

**Further Considerations**
1. Fixture freshness: live websocket ticks require event IDs that the feed currently publishes. Keep seeded Hive market snapshots as fallback, and refresh committed demo fixtures periodically if the team wants live ticks on every demo event.
2. Bloc interface implementation: validate early that Dart accepts demo blocs implementing existing concrete bloc classes while extending `Bloc<Event, State>`. If analyzer rejects a specific class, use a demo wrapper screen for that dependency rather than modifying production bloc constructors.
3. Demo persistence policy: recommended default is reset-on-Try-Demo for a predictable walkthrough, with an optional later toggle to preserve the user's previous local demo session.
